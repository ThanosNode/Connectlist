The Problem: Mismatched Authentication

Client-side (localStorage): The client (your React app) is storing some form of authentication information (e.g., a token, user ID) in localStorage. This allows the client to "remember" the user is logged in across page reloads.
Server-side (Sessions): The server is relying on traditional session-based authentication. This usually involves storing a unique session ID in a cookie and associating data with that session on the server.
The server doesn't know about the data in localStorage, and the client's localStorage doesn't automatically create a server-side session. Hence the "401 Not authenticated" errors.

The Solution: Consistent Authentication (Usually Token-Based)

The most common and recommended solution is to switch to a token-based authentication system, such as JSON Web Tokens (JWT). Here's the general approach:

Server-Side Changes:

Generate JWTs: When a user logs in successfully, the server generates a JWT. This JWT contains information about the user (e.g., user ID) and is digitally signed for security.
Send JWT to Client: The server sends the JWT back to the client in the login response.
Verify JWTs: For every protected API endpoint (like /api/listings), the server must now:
Extract the JWT from the request (usually in the Authorization header).
Verify the JWT's signature to ensure it's valid and hasn't been tampered with.
If the JWT is valid, extract the user information from it and allow the request to proceed.
If the JWT is invalid or missing, return a 401 error.
Remove Session Dependency: The server no longer needs to rely on sessions for authentication.
Client-Side Changes:

Store JWT: When the client receives the JWT from the server after login, it stores it in localStorage (or, more securely, in an HTTP-only cookie if possible).
Include JWT in Requests: For every protected API request, the client must include the JWT in the Authorization header (e.g., Authorization: Bearer <JWT>).
Handle Token Expiration: JWTs have an expiration time. The client needs to handle cases where the token has expired (e.g., by redirecting the user to log in again or by implementing a token refresh mechanism).
Remove localStorage Authentication: Remove any other authentication logic that solely relies on localStorage.
Why JWTs are Preferred

Stateless Authentication: The server doesn't need to store session information, making it more scalable.
Cross-Origin Compatibility: JWTs work well across different domains.
Security: JWTs are digitally signed, providing a good level of security.
Important Considerations

Security Best Practices: Implement JWT security best practices (e.g., using strong secret keys, setting appropriate expiration times, using HTTPS).
Token Storage: Storing JWTs in localStorage is vulnerable to XSS attacks. HTTP-only cookies are more secure but require server-side support for setting and reading them.
Token Refresh: Implement a mechanism to refresh expired tokens to improve user experience.
By implementing JWT-based authentication, you'll ensure that both your client and server are on the same page regarding user authentication, resolving the "401 Not authenticated" errors.
